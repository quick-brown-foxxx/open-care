## Findings — Ranked by Severity and Architectural Depth

### 🔴 CRITICAL (Must fix before production — trust or safety violations)

---

#### C1 — No available-balance check on disbursement writes

**Area:** Money/accounting — missing balance validation pattern.
**Files:** `apps/api-write/src/routes/disbursements.ts:83-106`

**What happens:** The disbursement write handler builds the payload and calls `appendLedgerEvent` with no check that `total_donations - total_disbursements >= new_disbursement_amount`. The `getTotals()` function exists but is never called by the write path.

**Impact:** An operator can record a disbursement that exceeds available donations. The ledger would show a negative available balance with no error, no alert, no warning. This undermines the trust foundation — donors see their money was spent but the math doesn't add up.

**Root cause:** The write side was built without the operational safety patterns that protect against operator mistakes. The read side computes totals correctly, but the write side doesn't use them as a guard.

**Fix pattern:** Before `appendLedgerEvent`, call `getTotals(db)`, compute `available = BigInt(totals.total_donations_usdc_minor) - BigInt(totals.total_disbursements_usdc_minor)`, and reject with `422 VALIDATION_ERROR` if `new_amount > available`.

---

#### C2 — No idempotency on disbursement writes

**Area:** Idempotency — missing duplicate-detection pattern.
**Files:** `apps/api-write/src/routes/disbursements.ts:52-136`

**What happens:** There is no idempotency key, no `receipt_ref` uniqueness constraint in the ledger, and no pre-insert duplicate check. An operator double-clicking "Submit" or a network retry could create two identical `disbursement_recorded` events.

**Impact:** The ledger would show double-spending of the same funds. The hash chain would be valid but semantically corrupt. Contrast with the webhook ingest path which has `ON CONFLICT DO NOTHING` on inbox PK and `checkDuplicateDonation` against the ledger — the operator write path has zero equivalent protection.

**Fix pattern:** Add an `idempotency_key` field to `DisbursementRequestSchema` (UUID generated by frontend). Check for existing ledger event with same `idempotency_key` before inserting. Or add a UNIQUE constraint on `receipt_ref` in `ledger_events` (via a partial index on `event_type = 'disbursement_recorded'`).

---

#### C3 — No idempotency on correction writes

**Area:** Idempotency — same pattern gap as C2.
**Files:** `apps/api-write/src/routes/corrections.ts:65-187`

**What happens:** The same `(corrects_sequence_no, replacement_fields)` combination can be submitted twice, creating two correction events for the same target. No duplicate check exists.

**Impact:** The bivalent API would show two correction events for the same original event, confusing verifiers. The hash chain remains valid but the correction semantics are broken.

**Fix pattern:** Same as C2 — add `idempotency_key` or check for existing correction on the same `corrects_sequence_no` with identical `replacement_fields`.

---

#### C4 — No bounded retry for stale anchor lock refresh (infinite loop risk)

**Area:** Retry/error classification — missing bounded-retry pattern.
**Files:** `apps/anchor-cron/src/lib/recovery.ts:47-59`, `anchor-pipeline.ts:114-118`

**What happens:** `attempt_count` is incremented on each stale lock refresh but never checked against a maximum. A transaction stuck in "confirmed" but never reaching "finalized" would cause the stale lock to be refreshed indefinitely — every cron tick and every manual trigger.

**Impact:** The `attempt_count` grows without bound, the lock is perpetually refreshed, and the anchor pipeline never makes progress for that head. This is a resource leak and a silent failure mode.

**Fix pattern:** Check `attempt_count >= MAX_RETRIES` (e.g., 10) before refreshing. If exceeded, mark the run as `failed` with `lock_expired_max_retries` and release the lock.

---

#### C5 — No dry-run/preview mode for operator writes

**Area:** Observability/operational — missing dry-run pattern for immutable ledger.
**Files:** Nowhere in the codebase.

**What happens:** The ledger is append-only and immutable. Once a disbursement or correction is written, it cannot be deleted — only corrected with another event. An operator who makes a typo in `amount_usdc_minor` or `receipt_ref` has no preview. The correction mechanism exists but is itself irreversible.

**Impact:** Operator mistakes become permanent ledger events requiring correction events to fix. A dry-run mode is the standard safety pattern for append-only systems — it lets the operator see the exact event that would be created, including the computed `event_hash`, before committing.

**Fix pattern:** Add `?dry_run=true` query parameter to write endpoints. When set, compute the full event (including `event_hash`) and return it in the response without inserting into the ledger.

---

#### C6 — `appendLedgerEvent()` is not concurrency-safe

**Area:** Append-only audit log — missing atomic read-check-write pattern.
**Files:** `packages/vault-db/src/helpers/ledger-append.ts:101-149`

**What happens:** The function reads the current head, computes `next_sequence_no = head.sequence_no + 1`, builds the hash preimage with that sequence number, then inserts **without explicitly inserting `sequence_no`**. SQLite autoincrements `sequence_no` independently.

**Race scenario:** Two concurrent writers both read head = N, both compute `next_sequence_no = N+1`, both compute hashes for sequence N+1. SQLite assigns actual sequence numbers N+1 and N+2. The second writer's row has `prev_hash` pointing to head N, but its actual sequence is N+2 — the chain is broken.

**Fix pattern:** Insert with explicit `sequence_no` value. On PK conflict, re-read head and retry. The existing retry loop already handles `event_hash` UNIQUE collisions; extend it to also handle `sequence_no` PK conflicts.

#### C7 — Anchor pipeline marks `published` before ledger append succeeds

**Area:** State machine — transition not atomic with source-of-truth write.
**Files:** `apps/anchor-cron/src/lib/anchor-pipeline.ts:218-245`, `apps/api-read/src/routes/verify.ts:87-102`, `apps/api-read/src/routes/health.ts:75-76`

**What happens:** Step 8 calls `clearLockOnSuccess()` (sets `anchor_runs.status='published'`). Step 9 calls `appendLedgerEvent()`. If Step 9 fails, the error is **logged but swallowed** (line 239-245). The function still returns `status: 'published'`.

**Result:** `anchor_runs` says published, but `ledger_events` has no `anchor_published` event. `/api/verify` reads anchors from `anchor_runs` (line 87-89), not from `ledger_events`. `/api/health` reads `anchor_runs` for staleness (line 75-76). A donor sees a "published anchor" in `/api/verify` that is **absent from the append-only public ledger**.

**This is a silent trust violation (STV1-STV3).** The system reports healthy while the chain and anchor are inconsistent.

**Fix pattern:** Reverse the order: append ledger event first (Step 9 before Step 8). If append fails, set `status='tx_confirmed_ledger_pending'` with tx_signature preserved for recovery. Update `/api/verify` and `/api/health` to cross-reference `anchor_runs` with `ledger_events` — only report anchors that have a matching `anchor_published` event.

---

#### C8 — Public verifier exits success when anchor verification is skipped

**Area:** Trust verification — purpose violation, fail-fast violation.
**Files:** `test/verify/verify-chain.ts:511-578`

**What happens:** When `/api/verify` returns no `latest_anchor`, the verifier produces `status: 'skip'`. The `ok` computation is `checks.every(c => c.status !== 'fail')` — skips are treated as non-failures. CLI exit code 0 for "all checks passed or were explicitly skipped."

**Impact:** A donor runs the verifier, gets exit 0, and believes the chain is verified against Solana. But no anchor comparison was performed. The verifier proved database consistency, not on-chain anchoring — which is its entire purpose.

**Additionally:** The verifier does not fetch the real Solana transaction memo. It trusts `/api/verify`'s reported `memo_text` and `anchored_head_hash`. A compromised or buggy API could return fake anchor data and the verifier would pass.

**Fix pattern:** Remove the `skip` status for anchor verification. Treat missing anchor as `fail`. Add Solana RPC fetch support (gated behind `--rpc-url` or `SOLANA_RPC_URL` env) to independently verify the on-chain memo. When RPC is unavailable, report "anchor metadata checked against API but on-chain verification skipped — use --rpc-url for full verification" and exit non-zero.

#### C9 — `redact()` is defined but NEVER CALLED — logging policy is convention-only

**Area:** Privacy — missing automated enforcement of logging safety.
**Files:** `packages/vault-core/src/logging.ts:296` (definition), zero call sites in any `apps/` or `packages/` source.

**What happens:** The `redact()` function has a comprehensive `DELETE_KEYS` set covering all secret and sensitive field names. But it is **never imported or called** in any production code. Every `logInfo`, `logWarn`, `logError` call site must manually avoid passing sensitive fields. There is zero automated enforcement.

**Impact:** A developer adding `logInfo('User registered', { telegram_user_id: userId })` would not be caught by any lint rule, test, or runtime check. The entire logging privacy policy is enforced by code review convention only. This is a ticking time bomb for privacy violations.

**Fix pattern:** Integrate `redact()` into the logging pipeline — either wrap `logInfo`/`logWarn`/`logError` to always pass through `redact()`, or add a CI lint rule that bans sensitive key names in log context objects.

#### C10 — Staging Workers expose internal routes via `workers_dev`

**Area:** Security — defense-in-depth failure, service binding trust model bypassed.
**Files:** `apps/tg-bot/wrangler.jsonc:1-49`, `apps/ingest/wrangler.jsonc:1-69`, `apps/operator/wrangler.jsonc`, `apps/api-read/wrangler.jsonc`

**What happens:** Four Workers have no top-level `workers_dev=false` (only in `env.production`). Wrangler defaults `workers_dev` to `true`. Internal unauthenticated routes are documented as "service binding only" but are reachable via `*.workers.dev`:

- `tg-bot`: `/tg/internal/pending-requests`, `/tg/internal/send-code` — zero auth
- `ingest`: `/internal/reconcile` — zero auth
- `operator`: auth-protected but extra attack surface
- `api-read`: public read routes only, lower impact

**Impact:** Attackers can list pending beneficiary requests, attempt code delivery, and trigger reconciliation scans — all without operator auth.

**Fix pattern:** Set `workers_dev: false` at top level for all Workers with internal-only routes. Add CI invariant test: for each Worker whose source contains routes not in its public `routes` array, assert `workers_dev === false`. As defense-in-depth, add auth middleware to internal routes even though service bindings should be the only ingress.

#### I1 — No reconciliation pattern between donations and disbursements

**Area:** Money/accounting — missing reconciliation alert.
**Files:** `packages/vault-db/src/helpers/queries.ts:186-212`, `apps/api-read/src/routes/health.ts`

**What happens:** `getTotals()` computes both sums independently but never compares them. The health check does not include an `available_balance_ok` field. The system can silently drift into negative available balance.

#### I2 — Anchor pipeline is a procedural if-else chain, not a state machine

**Area:** State machine — scattered failure handling across 3 files.
**Files:** `apps/anchor-cron/src/lib/anchor-pipeline.ts:104-256`, `lock.ts`, `recovery.ts`

**What happens:** `runAnchor()` is a 150-line function with 9 sequential steps, each guarded by `if (!result.ok) return { status: 'failed' }`. The lock protocol and recovery path are separate modules. States exist in the DB schema but the code doesn't model them as a state machine.

**Impact:** Adding a new failure mode requires understanding the entire procedural flow across 3 files. A state machine would make each state and transition explicit, testable in isolation, and safe against invalid transitions.

**Fix:** Extract explicit state machine with named states, defined transitions, and transition guards. Not launch-blocking but important for maintainability as failure modes grow.

#### I3 — Inbox processing is an if-else chain, not a state machine

**Area:** State machine — 6-state inbox with complex retry logic buried in branches.
**Files:** `apps/ingest/src/lib/inbox.ts:115-247`

**What happens:** `processInbox()` is a 130-line function with nested if-else branches handling retryable errors, permanent errors, parse failures, duplicates, and successful processing. States exist in schema but transitions are implicit.

**Fix:** Extract explicit state machine with named transition functions. Each transition testable independently.

#### I4 — Bot conversation `in_flight` state is never set

**Area:** State machine — documented state machine is misleading.
**Files:** `apps/tg-bot/src/lib/code-delivery.ts:94-195`, `apps/tg-bot/src/routes/internal/send-code.ts:40-146`

**What happens:** The documented state machine says `pending → in_flight → delivered/failed`, but the code never sets `in_flight`. The conversation goes directly to `delivered` or `failed`. Two operators sending code for the same conversation simultaneously could both succeed.

**Fix:** Set `in_flight` before delivery attempt. Check for `in_flight` to prevent concurrent delivery. Roll back to `pending` on failure.

#### I7 — No retry backoff in inbox processing

**Area:** Retry/error classification — bounded retries without delay.
**Files:** `apps/ingest/src/lib/inbox.ts:143-167`

**What happens:** When a retryable error occurs, the row stays in `received` status. The next attempt happens on the next poll (next webhook or cron tick) with no delay. Under rate limiting (429), the 10 retry attempts could be exhausted in minutes.

**Fix:** Add `retry_after_utc` field to inbox rows. Skip rows where `retry_after_utc > now()`. Set `retry_after_utc = now() + backoff` on retryable errors.

#### I8 — No fetch timeout on RPC calls

**Area:** External integration — hanging RPCs can consume full 30s CPU budget.
**Files:** `apps/ingest/src/lib/solana-rpc.ts:101`

**What happens:** Raw `fetch()` calls with no `AbortSignal.timeout`. A hanging RPC endpoint could block the Worker for up to 30 seconds.

**Fix:** Add `signal: AbortSignal.timeout(10_000)` to all outbound `fetch()` calls.

## Architectural Root Cause Summary

The system has a **strong read side and trust foundation** but a **weak write side**. The patterns that protect against operator mistakes — balance checks, idempotency, dry-run, bounded retries — are the ones most conspicuously absent. This asymmetry is the signature of a system where the cryptographic trust story (hash chain, canonical JSON, append-only triggers) was prioritized over the operational safety story (what happens when the operator makes a mistake?).

| Category                      | Strength                                                                                | Weakness                                                                                  |
| ----------------------------- | --------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| **Hash chain / cryptography** | Excellent — RFC 8785, normative test vector, Python cross-verification, collision retry | Concurrency race in `appendLedgerEvent` (C6)                                              |
| **Append-only enforcement**   | Excellent — SQLite triggers, single write path, bivalent corrections                    | Anchor pipeline can leave ledger event missing (C7)                                       |
| **Privacy architecture**      | Excellent — separate D1, HMAC refs, AES-GCM chat IDs                                    | Logging policy is convention-only — `redact()` dead code (C9)                             |
| **Webhook ingest**            | Excellent — idempotent inbox, error classification, ACK-fast                            | No fetch timeout (I8), no retry backoff (I7)                                              |
| **Operator write safety**     | **Critically weak** — no balance check (C1), no idempotency (C2, C3), no dry-run (C5)   | This is the biggest gap                                                                   |
| **Anchor pipeline**           | Good lock protocol, recovery path                                                       | Procedural if-else chain (I2), published-before-ledger (C7), no bounded retry (C4)        |
| **Public verification**       | Good hash recomputation                                                                 | Verifier doesn't independently check Solana (C8), exits 0 on skip                         |
| **Security boundaries**       | Good secret isolation, constant-time auth, service binding model                        | `workers_dev` exposes internal routes (C10), per-isolate rate limiter (I20)               |
| **State machines**            | States exist in DB schemas                                                              | No explicit state machines — procedural if-else chains in anchor, inbox, bot (I2, I3, I4) |
| **External integrations**     | Good Helius RPC anti-corruption layer                                                   | Telegram API not validated (I5), anchor-cron RPC errors not classified (I6)               |

## Trustworthiness Verdict

**Is this system trustworthy enough for production?**

**No, not yet.** The cryptographic trust foundation is solid, but three categories of gaps must be closed:

### Must fix before any production launch (10 Critical findings):

1. **C1-C3, C5:** Operator write safety — balance validation, idempotency, dry-run. Without these, operator mistakes become permanent ledger corruption with no guardrails.
2. **C6:** Ledger concurrency safety — the hash chain can be silently broken by concurrent writes.
3. **C7:** Anchor-ledger consistency — the system can report healthy while the chain and anchor are inconsistent.
4. **C8:** Verifier truthfulness — the verifier exits success when it didn't actually verify the anchor.
5. **C9:** Logging privacy enforcement — `redact()` must be integrated or a CI lint rule added.
6. **C10:** Staging security — `workers_dev=false` on all Workers with internal routes.
7. **C4:** Bounded retry for stale anchor locks — infinite loop risk.

### Should fix before production (Important findings):

- I1-I20 as listed above. The state machine gaps (I2-I4) are not launch-blocking but will make the system increasingly fragile as failure modes grow.

### What is already trustworthy:

- Hash chain mechanics, canonical JSON, normative test vector, Python cross-verification.
- Append-only enforcement (SQLite triggers, single write path).
- Privacy schema (separate D1, HMAC, AES-GCM).
- Webhook idempotency and error classification.
- Secret isolation and constant-time auth.
- Bivalent correction model.
- API contract adoption.
